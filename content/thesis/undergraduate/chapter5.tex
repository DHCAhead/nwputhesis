\chapter{实验结果与总结展望}
\chaptermark{实验结果与总结展望}

本章在 DeepCAD 子集上验证第~3、4 章给出的点级序列化与标准 next-token 自回归生成框架。其中 §5.1 描述实验环境与数据，§5.2 给出训练收敛性证据，§5.3 给出生成有效率主表并与毕业设计任务书 60\% 目标对比，§5.4 引入典型 B-rep 生成基线进行有效率对比，§5.5 给出三态分布并讨论失败案例，§5.6 总结局限，§5.7 给出结论与后续工作。

\section{实验环境与数据集}

\subsection{硬件与软件}

训练在校内 GPU 服务器上完成，单机配置 8 $\times$ NVIDIA H20 GPU（每张物理显存约 95\,GB，节点系统内存 1.0\,TiB）。本论文实际训练使用其中 4 张 H20，启动命令通过 \texttt{CUDA\_VISIBLE\_DEVICES=3,4,5,6} 显式锁定。软件栈在 Linux 节点上以 Python 3.10 为基础，深度学习框架使用 PyTorch 2.x\cite{paszke_2019_pytorch} 与 PyTorch Lightning 2.x（DDP 多卡并行），几何处理依托 OpenCASCADE Python 绑定，数据 I/O 使用 NumPy 与 h5py。

\subsection{数据集（DeepCAD 子集）}

本论文采用 DeepCAD 数据集\cite{wu_2021_deepcad} 的去重子集作为训练与评测数据来源。子集经第~3 章描述的预处理流程（STEP $\to$ 拓扑分解 $\to$ 面 / 边采样 $\to$ 量化 $\to$ 拓扑感知排序）转换为点级 token 序列：训练样本 64\,351 条、验证样本 6\,096 条，总样本数 70\,447，训练 : 验证 比例约为 91.3 : 8.7。数据词表规模 4\,053（50 面索引 + 4\,000 量化坐标 + 3 个结构 token），模型 \texttt{vocab\_size = 4054}（额外引入 PAD 用于批内对齐）。本论文未单独划分独立测试集，模型选择与最终汇报均基于上述验证集，由此带来的泛化偏差风险将在 §5.6 局限性中说明。

该数据划分适合原型阶段的可行性验证，但并不等同于严格的跨数据集泛化评估。训练集和验证集来自同一 DeepCAD 子集，零件类型、建模风格和预处理规则一致，因此验证损失更能反映模型对同分布样本的拟合情况。若部署到 ABC 全量数据、Fusion 360 真实设计记录或工业企业内部 CAD 库，模型可能遇到更多自由曲面、更复杂的装配边界和更长的拓扑链路，当前有效率不能直接外推。本文在结论中将结果表述为“当前实验设定下的工程可行性”，正是基于这一边界。

在以“可行性验证”为定位的前提下，DeepCAD 子集对本文闭环的关键环节仍具备足够的检验能力。它包含大量机械零件，面、边和实体结构相对清晰，预处理后的训练 / 验证规模达到约 7 万级样本，足以让 8 层、256 维的轻量 Transformer 拟合稳定的序列骨架。对于本科毕设而言，先在该子集上打通“数据转换 $\to$ 模型训练 $\to$ 推理采样 $\to$ OCC 重建 $\to$ BRepCheck 评估”五个环节、并保证每个环节都有可追溯的日志和参数记录，比一开始就追求全量多数据集更利于保证实验的可复现与结论可核查。后续若要扩展到 ABC 全量或工业 CAD 库，可以在不改变现有评测口径的前提下重训。

\section{训练收敛性分析}

\subsection{验证集交叉熵曲线}

按 $\text{PPL} = e^{\text{CE}}$ 换算，hr2\_patch1 最终困惑度约为 1.110，表明模型对验证序列中下一个 token 的预测分布已经接近确定性输出。两个 run 在数据序列化方式上完全一致，差异仅来自模型与正则配置，因此该对照仅用作说明“patch 化（含模型与正则调整）整体上对验证损失有显著影响”，不作为排序策略的消融证据。

需要注意，验证集交叉熵较低并不自动保证每个生成样本都能通过 BRepCheck。交叉熵按 token 平均，模型在大量容易预测的位置上表现很好，会拉低整体损失，但少量关键结构 token 的错误仍可能导致解析失败或拓扑不闭合。例如，SEP 位置偏移、面索引范围错误、END 提前出现等问题，在 CE 曲线上可能只占极少比例，却会对单个样本造成灾难性影响。因此，本文把训练收敛性视为生成有效性的必要但不充分证据，后续 §5.3、§5.4 和 §5.5 的重建评测才是最终工程指标。

从曲线形态看，hr2\_patch1 在 200 epoch 内没有出现明显验证损失反弹，说明当前模型容量和正则配置与数据规模大致匹配。hr2\_patch\_hreg2 的验证损失全程明显高于 hr2\_patch1，表明它使用的“分层正则化项”在当前数据规模下并未带来更好的 token 预测——这并不意味着分层正则一定有害，更可能是该正则项的强度需要更细的调参，但在没有更多 ablation 的情况下，本文不对其具体形式作强解释。这里只把 hr2\_patch\_hreg2 作为同数据、同训练周期下的对照 run 使用，用于说明最终选择 hr2\_patch1 作为主模型具有一份实测意义上的依据，而非完全凭超参直觉。两个 run 之间数据序列化方式完全一致，差异仅来自模型与正则配置，因此该对照仅说明“patch 化（含模型与正则调整）整体上对验证损失有显著影响”，并不构成对 §3 中拓扑感知排序或 §4 中任何具体组件的消融。本节因此把它列为“主模型选择依据”而非“组件贡献证据”。

\subsection{学习率调度与训练时长}

学习率随 step 推进的曲线由 201 行 step-level 记录刻画，整体呈现“warmup 上升 $\to$ 中段稳定 $\to$ 末段衰减”的三段式形态。hr2\_patch1 单次完整训练的端到端时长为 17 小时 28 分钟，按 200 epoch 均摊得平均每 epoch 约 5 分 14 秒。4 张 H20 通过 \texttt{torchrun --nproc\_per\_node=4} 启动 DDP 并行，训练日志记录了启动命令与 epoch 级 best-loss 序列。训练全程启用 AMP 混合精度与梯度检查点，单卡 95\,GB 显存给 $d_{\text{model}} = 256$、$\text{max\_seq\_len} = 4\,096$、per-GPU batch = 32 的训练负载留出充足容量，未观察到显存压力。

训练时长也说明本文方法在工程上具备一定可迭代性。一次 200 epoch 训练控制在一天以内，意味着研究者可以在同一硬件条件下尝试少量结构或超参数变体。若单次训练需要数天甚至数周，本科毕设阶段很难完成完整对比。当前模型规模较小，训练成本主要来自长序列注意力和数据 I/O，而不是参数量本身。后续若扩大到全量 ABC 或更长上下文，需要重新评估 batch size、梯度累积和分布式策略。

\section{生成与重建有效性分析}

\subsection{采样配置与样本数}

本节用两个独立的生成 run 评估反向重建有效性：

\begin{itemize}
    \item hr2\_patch1（Epoch~200 best checkpoint）：按 top-k = 1、top-p = 0.9 采样 1\,005 条 token 序列，每条经反向重建管线得到一个候选 Solid。本次评测 1\,005 个采样全部成功写出 STEP 文件（重建错误数为 0）。
    \item 早期 hr2 基线（无 patch 后缀）：在相同 top-k = 1、top-p = 0.9 配置下采样 3\,332 次，其中 3\,000 次成功写出 STEP 文件（332 次在重建阶段失败）。
\end{itemize}

两个 run 使用同一份评测脚本的五标量输出（序列总数 / 重建成功 / 重建失败 / 有效 BREP / 无效 BREP），保证口径一致。

hr2\_patch1 在 1\,005 个保存样本上的有效率为 980 / 1\,005 = 97.51\%。早期 hr2 基线在 3\,000 个保存样本上的有效率为 2\,711 / 3\,000 = 90.40\%。两者在“of saved”口径下的差值为 7.11 个百分点。该差值来自两个 run 在模型与训练配置上的整体差异，本文不将该差值单独归因于第~3 章的拓扑感知排序、第~4 章的具体网络结构调整或某一个具体组件。

从工程角度看，97.51\% 的 of saved 有效率说明：在已经成功写出 STEP 文件的样本中，绝大多数实体能够通过 OCC 的几何拓扑检查。结合 hr2\_patch1 的 0 次重建失败，可以进一步说明该 run 不仅 BRepCheck 通过率较高，而且序列解析和重建前端也较稳定。与早期 hr2 基线相比，hr2\_patch1 的提升不仅体现在有效率从 90.40\% 到 97.51\%，也体现在采样 1\,005 次全部成功写出 STEP。这一点在 §5.5 的三态分布中会更直观。

不过，两组 run 的样本数不同：早期 hr2 基线为 3\,332 次尝试、3\,000 个保存样本，hr2\_patch1 为 1\,005 次尝试且全部保存。因此，本文把二者作为工程进展对照，而不是严格统计检验意义上的同条件比较。若要更严谨评估提升幅度，应在同一采样数量、同一随机种子策略、同一 checkpoint 选择规则和同一后处理版本下做配对实验。本文当前结论限定为：hr2\_patch1 在已完成的大样本评测中达到 97.51\% of saved，有效超过任务书目标。

从 of attempts 口径看，hr2\_patch1 的 980 / 1\,005 同样为 97.51\%，因为该 run 没有 S1 重建失败。早期 hr2 基线则为 2\,711 / 3\,332 = 81.36\%，低于其 of saved 口径。两个口径的差异说明，评价生成式 CAD 系统时不能只看已保存样本中的有效比例，还要关注有多少采样结果根本无法写出 STEP。对于实际使用者而言，无法产出文件和产出无效文件都会带来额外筛选成本。因此，S1 的下降与 S3 的提升同样重要。hr2\_patch1 在这两方面都优于早期基线，是本文认为其工程稳定性更好的主要依据。

还需要说明的是，top-k = 1 与 top-p = 0.9 的组合整体偏向保守采样。保守采样通常会减少随机性，生成更接近训练分布高概率区域的样本，从而有利于提高有效率，但它可能牺牲多样性，使输出集中在较常见的拓扑模式上。本文主目标是完成任务书要求的有效 B-rep 生成闭环，因此优先采用稳定配置。若后续研究关注设计多样性或新颖性，应系统比较不同温度、top-k、top-p 下的有效率、多样性和几何复杂度，而不能只在单一采样配置下下结论。

\subsection{与任务书目标的对比}

毕业设计任务书在评测维度上对生成模型给出“有效率不低于 60\%”的红线指标。该数字并非来自某一具体文献，而是任务书制定方综合考虑了 B-rep 生成原型系统的工程门槛、毕设阶段的可控复杂度以及 CAD 内核校验的严格程度后给出的工程下限——它确保通过该门槛的系统至少能批量产出多数样本可用的实体，但并不要求达到工业部署级的稳健性。在此基线下，本文早期 hr2 基线的 90.40\% 与 hr2\_patch1 的 97.51\% 均显著超过该指标，分别高出 30.40 与 37.51 个百分点。这一结果支持以下结论：在当前实验设定下，本文提出的点级序列化方案与标准 next-token 自回归生成路线在 BRepCheck 通过率这一维度上是工程可行的，可以为后续在更大规模数据集上的研究提供一个可用的工程起点。

需要强调的是，上述结论严格限定在当前实验设定（DeepCAD 子集、给定的采样配置与单次评测）之下成立，不能解读为本文方法在所有 CAD 数据集上的通用泛化能力。具体的边界条件包括但不限于：训练 / 验证分布同源、零件复杂度受 \texttt{max\_seq\_len = 4\,096} 上限过滤、采样配置固定为 top-k = 1 与 top-p = 0.9 的保守组合。相关边界在 §5.6 局限性中作进一步明确，本节只做工程指标层面的对照陈述。

此外，任务书“60\%”是工程指标而非学术竞赛指标。超过 60\% 说明本文原型系统达到了毕业设计要求，但并不意味着模型已经具备工业生产环境所需的鲁棒性。真实 CAD 流程通常还会要求尺寸约束、装配关系、可编辑设计历史、可制造性检查和用户条件控制等更高维的能力。本文当前只证明了“无条件生成的 B-rep 候选在当前子集上可以较高比例通过基础几何检查”这一较窄的命题，距离“按设计需求生成可编辑零件”仍有明显距离。后者更可能由条件生成、约束求解以及交互式编辑等下一阶段工作共同承担。

\section{基线方法对比实验}

S2 类型的 25 个无效样本主要由 OCC 的 BRepCheck 报告标记为 invalid，常见原因包括 Shell 含游离边、面参数域与边曲线不匹配、面取向不一致等几何—拓扑细节问题。本论文未对 S2 样本作进一步的关键字级失败原因细分，相关后处理与修复管线作为后续工作（见 §5.7）。

三态分布还揭示了不同失败层次的改进方向。S1 失败通常意味着生成序列无法被解析为完整面段和边段，或 NURBS 拟合、Solid 构造阶段直接异常，这类问题更接近序列语法和前端重建稳定性。S2 失败则说明样本已经成功写出 STEP，但几何—拓扑一致性仍有问题，这类问题更接近局部曲面 / 曲线拟合精度、边界缝合容差和面取向修复。hr2\_patch1 的 S1 为 0，说明最前端可解析性已较好。剩余 25 个 S2 样本提示后续应重点改进边曲线与相邻面的一致性、ShapeFix 策略和失败样本分类。

由于本文未进行逐样本可视化分类，暂不把 S2 进一步拆成定量比例。更细的失败分析可以按照 OCC 错误码、几何位置和外观类型三条线展开：第一，统计 BRepCheck 返回的 NotClosed、InvalidCurveOnSurface、BadOrientation 等类别。第二，定位错误集中在边、面还是 Shell。第三，将典型失败样本与有效样本并排可视化，观察是否存在短边、薄壁、孔洞或高曲率区域。这样的分析能为后处理修复和下一轮模型改进提供更明确方向。

若进一步结合第~4 章的逆向重建流程，S2 样本可以被理解为“生成序列已经具备基本语法，但连续几何和拓扑闭合尚未完全一致”的边界案例。它们通常比 S1 更有研究价值，因为 S1 往往只能说明格式或拟合前端失败，而 S2 已经完成 STEP 写出，说明模型至少生成了较完整的面段和边段。对这类样本进行局部修复实验，可以评估无效样本是否具备可回收性。若少量容差调整或 ShapeFix 策略即可使其通过 BRepCheck，说明模型输出与合法实体之间的距离较近。若需要大幅修改面索引或重建曲线，则说明问题仍在生成模型本身。

由此，后续的失败案例分析不应只停留在统计数量上，而应形成“错误码—几何位置—可修复程度”的可执行清单——其中错误码反映内核为何判无效（如 \texttt{NotClosed} / \texttt{InvalidCurveOnSurface} / \texttt{BadOrientation}），几何位置反映错误集中在 Face、Edge、Wire 还是 Shell 层级，可修复程度反映后处理（如 ShapeFix 容差调节、Sewing 容差扩大、面取向自动反转）能否在不大幅修改模型输出的前提下使其通过 BRepCheck。把当前 2.49\% 的剩余无效样本按这三个维度拆开，就能把“还剩多少不合法”这一单一百分比转化为可行动的问题清单，从而指明优先级：是应该首先改进模型本身、调整采样策略、增强 NURBS 拟合的鲁棒性，还是强化拓扑修复管线。这是一项不需要重训模型即可完成的分析，本文已将其列入 §5.7 后续工作。

\section{局限性}

本论文在以下五个维度存在明确局限。

第一，训练与评测均限定在 DeepCAD 子集上，未在 ABC 数据集或 Fusion 360 子集上验证跨数据集泛化能力。第二，训练 / 验证集按 91.3 : 8.7 划分但未单独保留独立测试集，最优 checkpoint 与最终有效率均基于验证集评估，模型选择偏差风险在小数据集上尤为明显。第三，虽然本文纳入了多种 B-rep 生成基线的有效率对比，但各方法在采样数量、数据过滤和后处理版本上并非完全一致，因此该对比主要用于说明指标量级，而不是严格配对统计检验。第四，本论文未对第~3 章的拓扑感知排序策略以及第~4 章的 CompoundPointDecoder 等可选组件做受控对照消融，因此对其各自的独立贡献不作定量声明。第五，本论文主要以 BRepCheck 通过率为评估指标，未引入 Chamfer Distance、Coverage、Minimum Matching Distance 等几何相似度指标，几何细节质量的评估手段相对单一。

除上述五点外，本文还存在两个工程层面的限制。其一，当前序列化方案对复杂零件的压缩能力有限，面数和边数增加会直接推高序列长度，超过 4\,096 上限的样本会被过滤。因此，模型更擅长当前子集中的中等复杂度实体，对复杂装配件或大量小特征零件的覆盖不足。其二，当前生成是无条件的，用户无法指定零件类别、尺寸范围、功能约束或局部拓扑要求。对于实际 CAD 辅助设计而言，无条件生成只能提供候选形状，距离“按设计需求生成可编辑零件”仍有明显距离。

这些局限并不否定本文结果，而是明确其适用范围。本文的核心贡献是证明点级 NURBS 序列化与标准自回归模型可以打通 B-rep 生成闭环，并在任务书给定指标上取得足够高的有效率。下一阶段研究应在保持该闭环可复现的基础上，逐步补齐独立测试集、消融实验、多指标评价和条件控制，而不是一次性叠加过多未验证模块。

从论文结论表达上看，本文需要避免两类过度推断。其一，不能把 97.51\% 的有效率解释为“模型已经能够稳定生成任意 CAD 零件”，因为当前样本来自同一 DeepCAD 子集，且复杂度受序列长度过滤限制。其二，不能把 hr2\_patch1 相对早期基线的提升完全归因于单个组件，因为两个 run 之间存在模型配置、训练细节和后处理版本等综合差异。本文在正文中反复强调“当前实验设定下”和“工程进展对照”，正是为了让结论与证据强度相匹配。

不过，受限并不意味着结果缺乏价值。生成式 CAD 研究从想法到可运行闭环之间存在较高工程门槛：数据要能从 STEP 转成可训练序列，模型要能在长序列上训练收敛，采样结果要能被解析为合法骨架，重建结果要能写回 STEP，最终还要经几何内核检查——任何一个环节失败都意味着整个链路无法落地。本文已经打通了这条链路，并通过三态统计给出了可独立复核的有效性证据：1\,005 个采样样本全部成功写出 STEP、其中 980 个通过 BRepCheck，对应的脚本、日志、checkpoint 与配置均可追溯到 §5.1.3 的复现表。在这一基础上，后续工作只需沿着同一评测口径逐步增加数据规模、消融实验和条件控制，就可以在不推翻现有框架的前提下继续提高研究深度——这种“扩展式改进”的可承接性，本身就是工程闭环的一部分价值。

\section{总结与后续工作}

\subsection{工作总结}

本论文围绕“不依赖显式 VQ-VAE 码本、端到端可重建为合法 STEP 文件”这一总体目标，开展了三方面工作。(1) 设计了面向 B-rep / NURBS 的点级量化与窗口化序列表示方法，给出了 face\_block = 49 与 edge\_block = 14 的序列骨架、4000 级标量均匀量化、数据侧 4053 维统一词表、模型侧 4054 维输出词表与拓扑感知排序的具体方案。(2) 实现并训练了 LLaMA3 风格的标准 next-token 自回归生成模型，主干由 RoPE、SwiGLU、RMSNorm 与 GQA 组成，输出头为“主干 norm $\to$ \texttt{ar\_lm\_head}”。(3) 实现了基于 OpenCASCADE 的逆向重建与 BRepCheck 评估闭环，构成一条可独立运行的工程流水线。

\subsection{初步结论}

在 DeepCAD 子集上，hr2\_patch1 经 200 个 epoch 训练后验证集交叉熵损失收敛至 0.10486，对应困惑度约 1.110。在 1\,005 个独立生成样本上，全部成功写出 STEP 文件、其中 980 个通过 BRepCheck，of saved 与 of attempts 口径下的有效率均为 97.51\%。早期 hr2 基线在相同采样配置下经 3\,332 次采样得到 3\,000 个 STEP、通过 2\,711 个，对应 of saved 90.40\%、of attempts 81.36\%。两组结果均明显高于毕业设计任务书“60\%”的红线指标。这一现象在当前实验设定下可以支撑以下初步结论：本文提出的“点级 NURBS 序列化 + 标准 next-token 自回归 + OCC 逆向重建”工程路线是可行的，可作为后续在更大规模、更多样化 CAD 数据上开展研究的工程起点。

需要重申的是，本结论既不直接外推到跨数据集泛化，也不应解读为本文方法在所有条件下严格优于 VQ-VAE / 扩散路线。同时也不把 hr2\_patch1 相对早期基线的提升归因于本文任何一个具体组件——拓扑感知排序、CompoundPointDecoder、patch 化中的具体网络与正则调整等都未做独立受控消融，相应的 ablation 已在 §5.7.3 后续工作中列出。

\subsection{后续工作}

针对 §5.6 列出的局限，本文后续可在如下方向继续展开：(a) 启动“DFS + 最大索引排序”与随机排序的受控对照消融，定量考察拓扑感知排序的独立贡献。(b) 启用 $W > 1$ 的窗口化分支并实测 CompoundPointDecoder 对生成质量的影响，与本文 $W = 1$ 的主分支做配对对比。(c) 在全量 DeepCAD 与 ABC 等更大规模数据集上重训，观察大规模数据下的收敛与泛化。(d) 在统一采样数量、统一随机种子策略和统一评估脚本下重跑各基线方法，形成更严格的横向比较。(e) 引入 Chamfer Distance、Coverage、Minimum Matching Distance 等几何相似度指标，丰富生成质量评估。(f) 对 S2（产出但无效）样本接入 OCC ShapeFix 等后处理修复管线，量化失败样本的可回收比例。

在上述六个方向中，本文建议把“排序消融”与“失败案例分类”列为最优先：前者可以直接检验第~3 章最关键的方法设计是否带来独立收益，结果不论正负都能让“拓扑感知排序”从设计选择落实为可量化的方法贡献。后者只依赖现有 1\,005 个生成样本与 OCC 错误码，不需要重新训练模型，却能把当前剩余的 2.49\% 无效样本拆成可行动的修复优先级。二者都不改变整体研究路线，却可以显著提升论文结论的解释力与可推广性。若后续计算资源允许，再扩展到条件生成、跨数据集训练和几何相似度评估，将使本文原型从“可行性验证”逐步演化为更接近真实 CAD 辅助设计的系统。
